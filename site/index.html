Why AgenticKVM

AI agents are useful until the machine falls off the normal path.

Agents can reason, write code, and call APIs. They struggle when SSH is down, the OS is broken, a machine is stuck at BIOS, the browser session is unavailable, or recovery requires console-level access.

AgenticKVM is designed to put a safer control layer between agent intent and machine access. Every action is translated into a capability request, checked by policy, approved when required, executed only through registered providers, and recorded in audit.

How it works

Policy, approvals, audit, and provider registries before execution.

  1. Agent or tool request
  2. MCP or host adapter
  3. Target and provider registry
  4. Capability request
  5. Policy decision
  6. Approval if required
  7. Provider adapter
  8. Audit event
  9. Structured result

Safety guardrails

The interface is not the authority. Policy is.

Fail closed

Unknown capabilities, providers, targets, mappings, and malformed requests deny by default before provider execution.

Registered scope

Providers and targets must be explicitly configured. Request-supplied names are untrusted until validated by registries.

Approval is first-class

`approval_required` is a structured outcome, not an implicit grant. Gated actions wait for an explicit matching approval.

Full Control still has limits

Full Control can reduce prompts for in-scope actions, but it does not bypass audit, scope, emergency stop, redaction, or invariants.

Audit trail

Local audit scaffolding is redacted and hash-chain/checkpoint ready. Production audit-store decisions remain explicit gates.

Mock-only CI

Real providers are disabled by default. CI uses mocks, fixtures, contracts, and offline tests instead of live infrastructure.

MCP and agents

MCP tools route through the control plane.

AgenticKVM includes MCP-style models, a router, a dependency-free mock-only MCP adapter, and a mock-only host compatibility layer. These interfaces are translators, not authority boundaries.

The mainline project does not adopt a live MCP server. A Python MCP SDK trial exists separately and remains gated by human review; it is not a mainline dependency. Any future live MCP server must preserve provider registry, target registry, policy, approval, audit, and redaction behavior.

Getting started

Start with docs and mock-only tests. No live hardware by default.

Clone and inspect

git clone <repository-url>
cd AgenticKVM

Run the test suite

uv run --offline --with pytest --python python3.13 python -m pytest

Tests are designed for mocks, fixtures, and offline contracts. Do not point AgenticKVM at real hardware, credentials, or live provider endpoints unless a future operator-approved smoke plan explicitly says to do so.
